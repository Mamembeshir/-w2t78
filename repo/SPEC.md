Set up a Warehouse Intelligence & Offline Crawling Operations Platform that enables procurement analysts, inventory managers, and system administrators to collect product and supplier data from multiple internal sources, keep inventory accurate across facilities, and notify staff of exceptions without relying on any internet connectivity. The React web UI provides role-based navigation with dynamic route loading so an Inventory Manager can scan or enter barcode/RFID identifiers, receive stock into specific warehouse/bin locations, issue stock to work orders, transfer between warehouses, and record cycle counts with guided variance confirmation. A Procurement Analyst uses a configuration center screen to define and test collection rules per source, including URL patterns, parameters, pagination behavior, request headers, rate limits (for example, default 60 requests/minute per source), and offline anti-bot strategies such as rotating predefined user-agent strings and honoring crawl delays; a visual debugger shows the last 20 request/response samples captured locally with redacted secrets. Rule changes require a version note, can be canary-released to 5% of tasks for 30 minutes, and can be rolled back to the prior version in one click if error rate exceeds 2%. In-app notifications appear in a unified inbox with filters and read receipts, supporting subscriptions like "Safety stock breach," "Cycle count variance over $500.00," and "Crawl rule canary failed," plus scheduled digests at 6:00 PM; because the environment is offline, messaging is delivered in-app and to locally hosted SMTP/SMS gateways only when present, with no dependency on external providers.

The backend uses Django to expose REST-style APIs to the React frontend, enforce permissions, and orchestrate distributed crawling and warehouse operations, with MySQL as the system of record for users, roles, warehouses/bins, items, lots/serials, crawl sources/rule versions, tasks, quotas, and message logs. Crawling runs on horizontally scalable worker nodes that pull sharded tasks from a queued scheduler with priorities and per-source quotas; tasks are idempotent via deterministic request fingerprints, retries use exponential backoff (10s, 30s, 2m, 10m) with a max of 5 attempts, and resumable crawling persists checkpoints every 100 pages so work can continue after a node restart. Real-time quota and concurrency control must be strongly consistent: quota deductions and releases occur within a database transaction using row-level locking, with timeout holds released automatically after 15 minutes to avoid resource oversubscription, and waitlisted tasks auto-promote within 5 seconds when capacity frees up. Inventory accounting supports FIFO and moving-average costing per SKU, flags slow-moving stock when no issues occur for 90 days, and triggers safety-stock alerts when available quantity drops below the configured threshold for 10 consecutive minutes to prevent flapping. Security is local-first: username/password login only, passwords hashed with Argon2, sensitive fields such as supplier credentials and rule secrets encrypted at rest, and logs mask tokens by default; end-to-end request tracing, standardized error codes, pagination, and caching are implemented entirely on-prem, with send logs, retry records, and administrative audit trails retained for 365 days.

Everything must work 100% offline on local network.
